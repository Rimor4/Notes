# 云计算与虚拟化技术

### 1. 大数据与云计算

1. 新摩尔定律的含义是什么？

- **核心定义**：新摩尔定律（New Moore's Law）是云计算时代的产物。它指出：**每 18 个月，计算机的计算能力和传输带宽将翻一番，而价格则会下降一半。**
- **关键点**：与传统摩尔定律侧重“集成电路上晶体管数量”不同，新摩尔定律更强调**算力交付的经济性**。这意味着在云计算模式下，用户可以用越来越低的成本获得越来越强的处理能力。

------

2. 云计算有哪些特点？

云计算具有以下五个典型的压倒性特征（NIST 标准定义）：

- **按需自助服务**：用户可以根据需要自动获取算力、存储等资源，无需与服务商进行人工交互。
- **随时随地访问**：通过互联网（标准接入机制），用户可以使用手机、笔记本等各种终端访问云端资源。
- **资源池化**：服务商将计算资源汇聚成池，通过多租户模式，根据需求动态分配给不同用户。
- **快速弹性伸缩**：资源可以迅速、弹性地提供（上线）和释放（下线），让用户感觉资源是“无限”的。
- **精确计量计费**：云系统能够自动监控和优化资源使用，并根据实际消耗量（按流量、按时长等）向用户收费。

------

3. 云计算按照服务类型可以分为哪几类？

这是考试中的常考点，即云计算的 **SPI 三层模型**：

- **IaaS（基础设施即服务）**：
  - 提供底层的物理资源虚拟化（计算、存储、网络）。
  - 用户需要自己管理操作系统和应用程序（如：阿里云 ECS、AWS EC2）。
- **PaaS（平台即服务）**：
  - 提供开发、运行和管理应用的平台。
  - 用户无需关心底层硬件和 OS，只需关注应用开发（如：Google App Engine）。
- **SaaS（软件即服务）**：
  - 直接通过网络提供现成的软件应用。
  - 用户直接使用，无需任何安装和开发（如：钉钉、腾讯会议、Salesforce）。

------

4. 云计算技术体系结构可以分为哪几层？

根据讲义中的“云计算实现机制”部分，其技术体系结构通常分为四层：

1. **物理资源层**：包括原始的硬件资源（服务器、交换机、硬盘等）。
2. **资源池层（虚拟化层）**：通过虚拟化技术将物理硬件抽象化，形成计算池、存储池和网络池，实现资源的灵活分割和整合。
3. **管理中间件层**：云计算的“神经中枢”。负责资源管理、任务调度、安全监控、计费管理以及副本容错管理。
4. **SOA 服务接口层**：通过 Web Services 等接口，将云端能力封装成标准的服务提供给最终用户或第三方开发者。

### 2. Google云计算原理与应用

1. Google 云计算技术包括哪些内容？

Google 云计算技术是一个完整的技术栈，主要包括：

- **分布式文件系统 GFS**：底层存储。
- **分布式并行计算模型 MapReduce**：数据处理。
- **分布式锁服务 Chubby**：一致性协调。
- **分布式结构化数据表 Bigtable**：列式数据库。
- **分布式存储系统 Megastore**：介于关系型与 NoSQL 之间。
- **监控基础架构 Dapper**：分布式链路追踪。
- **交互式分析工具 Dremel**：海量数据秒级查询。
- **内存大数据分析系统 PowerDrill**：极致性能分析。
- **Google App Engine (GAE)**：应用程序托管平台。

2. 当前主流分布式文件系统有哪些？各有什么优缺点？

- **GFS (Google File System)**：
  - **优点**：高度容错，适合大规模顺序读写和追加操作，成本低（普通硬件）。
  - **缺点**：单 Master 存在瓶颈，不适合处理大量小文件。
- **HDFS (Hadoop Distributed File System)**：
  - **优点**：开源，生态极其成熟，高吞吐量。
  - **缺点**：延迟高，不支持并发写和随机修改。
- **Lustre/Ceph**：
  - **优点**：高性能，支持 POSIX 标准。
  - **缺点**：配置维护复杂，对网络要求高。

3. GFS 采用了哪些容错措施来确保整个系统的可靠性？

- **副本策略**：每个数据块（Chunk）默认有三个副本，分布在不同的机架。
- **Master 容错**：通过操作日志和检查点（Checkpoint）记录状态；Master 失效后由影子 Master 快速接管。
- **数据一致性检查**：使用校验和（Checksum）检测数据损坏。
- **ChunkServer 故障恢复**：Master 周期性发送 Ping 探测 ChunkServer 状态，发现故障则重新复制数据。

4. MapReduce 与传统的分布式程序设计相比有何优点？

- **易用性**：程序员只需编写 Map 和 Reduce 两个简单函数，无需关心分布式底层细节（如网络、并发）。
- **高可靠性**：自动处理节点失效，重新执行失败的任务。
- **可扩展性**：可以轻松扩展到成千上万个节点处理 PB 级数据。
- **数据本地化**：尽量在存储数据的节点上进行计算，大幅节省网络带宽。

5. 阐述 Bigtable 的数据模型和系统架构

- **数据模型**：一个稀疏的、分布式的、持久化的、多维有序映射表。由 **行键 (Row Key)**、**列键 (Column Key)** 和 **时间戳 (Timestamp)** 共同定位一个数据项。
- **系统架构**：
  - **客户端**：通过库与 Master 和子表服务器通信。
  - **主服务器 (Master)**：分配子表、负载均衡、垃圾回收。
  - **子表服务器 (Tablet Server)**：管理一组子表，处理数据的读写请求。
  - **底层支撑**：依赖 GFS 存储 SSTable，依赖 Chubby 进行元数据管理和选举。

6. 分布式存储系统 Megastore 的核心技术是什么？

其核心技术是**基于 Paxos 算法的同步复制机制**。

- 它在多个数据中心之间同步日志，实现跨地域的数据强一致性（Entity Groups 内部）。
- 结合了关系型数据库的语义（事务支持）和 NoSQL 的可扩展性。

7. 相比于行存储，列存储有哪些优点？

- **数据压缩率高**：同一列数据类型相同，更有利于压缩。
- **查询性能优**：查询时只需读取相关的列，避免了读取无关列的 I/O 浪费（极适合 OLAP 场景）。
- **扩展性强**：添加新列更加灵活，不影响旧有数据的存取。

8. 为什么 MapReduce 不适合实时数据处理？

- **高延迟**：Map 和 Reduce 阶段之间存在 Shuffle（排序和磁盘 I/O），必须等待所有 Map 完成才能开始 Reduce。
- **非长驻进程**：每次启动任务都有较大的初始化开销。
- **批处理导向**：设计初衷是为了处理大规模静态数据集，而非低延迟的流式响应。

9. Google App Engine 有哪些不同的开发环境？各自有什么特点？

- **标准环境 (Standard Environment)**：
  - **特点**：运行在受限的沙盒中，启动极快（秒级），支持自动缩减到零（节省成本），但对库和网络访问有严格限制。
- **柔性环境 (Flexible Environment)**：
  - **特点**：运行在 Docker 容器中，支持自定义运行环境（如安装特定库），支持 SSH 连接，能使用更高性能的 CPU 和内存，但启动速度较慢且不支持缩减到零。

### 3. Amazon 云计算 AWS

1. Dynamo 架构的主要技术

Dynamo 是 Amazon 核心的基础存储架构，为了满足高可用性、可扩展性和容错性，它采用了多种分布式系统领域的关键技术：

- **数据均衡分布：一致性哈希（Consistent Hashing）**
  - **原理**：将存储节点和数据的 Key 都映射到一个首尾相连的哈希环上。
  - **优点**：当增加或删除节点时，只影响环上相邻的节点，最大限度地减少了数据迁移量。此外，引入“虚拟节点”确保了物理节点间负载的均衡。
- **高可用性保障：数据复制（Replication）**
  - **机制**：数据被复制到哈希环上顺时针方向的 $N$ 个后续节点上，这 $N$ 个节点构成一个“首选列表”。
- **临时故障处理：数据回传机制（Hinted Handoff）**
  - **原理**：当某个节点 A 暂时不可用时，本应发往 A 的数据会发送到备份节点 B，B 会在本地临时存储这部分数据并附带一个“提示”。一旦 A 恢复正常，B 会将数据回传给 A。
- **跨数据中心容错：基于“偏好列表”的成员身份管理**
  - 确保数据的副本分布在不同的可用区域（AZ）或地理区域，防止单点故障。
- **最终一致性处理：向量时钟（Vector Clocks）**
  - **作用**：Dynamo 允许数据的多个版本同时存在。向量时钟用于捕获数据版本间的因果关系，从而在读取时发现并解决冲突（即“语法上的冲突合并”）。
- **永久故障修复：逆熵机制（Anti-entropy with Merkle Trees）**
  - **原理**：利用 Merkle 树（哈希树）快速比较不同节点间数据的差异，只同步不一致的部分，大幅降低网络流量。
- **成员身份与故障检测：Gossip 协议**
  - **原理**：节点之间像“传播流言”一样随机交换信息，从而在去中心化的环境下快速感知整个集群的成员变化和节点故障状态。

4. 地理区域和可用区域有哪些区别？

在 AWS 的架构中，这两个概念共同构成了其全球基础设施的基础：

- **地理区域（Region）**：
  - 是指 AWS 在全球范围内划分的独立地理区域（如美国东部、欧洲、亚太地区等）
  - 每个区域都是完全独立的，目的是为了降低延迟（更靠近用户）并满足法律合规性要求2。
- **可用区域（Availability Zone, AZ）**：
  - 每个地理区域内部包含多个相互隔离的可用区域3。
  - 一个可用区域通常由一个或多个物理数据中心组成，拥有独立的电源、冷却和网络设施4。
  - **区别点**：地理区域是物理上的远距离分隔；而可用区域通过区域内的低延迟光纤网络互联，主要用于实现应用的高可用性和容灾 5。

9. 简述 Share-Nothing 架构的特点。

AWS 的关系数据库服务 RDS（特别是 MySQL 集群）采用了 Share-Nothing 架构，其核心特点包括：

- **资源独立性**：每台数据库服务器都是完全独立的计算机系统，通过网络相连，不共享 CPU、内存或硬盘资源 6。
- **高可扩展性**：由于没有共享资源带来的争用瓶颈，当数据库处理能力不足时，可以通过简单地增加服务器数量来线性提高性能 7。
- **高并发支持**：多个独立的服务器增加了数据库并发访问的能力 8。
- **局限性**：对管理人员要求极高，需进行合理的“表单划分（Sharding）”，否则跨服务器查询会导致性能下降 9。

10. 简述 AWS 如何支持在线数据流应用。

AWS 通过以下服务的组合实现对在线数据流的支持：

- **消息传递与解耦**：利用**简单队列服务 SQS** 10。它在云计算平台的不同组件之间提供通信支持，允许数据流在不同处理环节中排队等待，确保系统在高负载下的稳定性 11。
- **存储与缓存**：使用 **DynamoDB** 提供低延迟的键值对存储 12，或者通过 **S3** 进行大数据量的流式转储 13。
- **内容推送**：通过 **CloudFront** 内容推送服务 14。它在边缘节点缓存数据副本，利用智能路由将请求分配给最近的节点，从而支持视频流、实时动态内容的高速分发 15。

11. 描述一个基于 Amazon AWS 的应用实例。

讲义中提到了 AWS 的典型应用场景（如 **Animoto** 案例）：

- **应用背景**：Animoto 是一个在线视频制作网站，面临极度不稳定的用户访问量。
- **AWS 的应用**：
  - 使用 **EC2** 实例处理视频渲染任务 16。
  - 结合 **CloudWatch** 监控实例状态，并通过 **Auto Scaling（自动缩放）** 功能在需求高峰期无缝增大计算能力，在低谷期自动缩小规模以降低成本 17。
  - 使用 **S3** 存储用户的原始图片和生成的视频文件 18。
  - 利用 **SQS** 管理待处理的任务队列，确保视频渲染任务有序执行 19。
- **核心价值**：这种架构让初创企业无需购买昂贵的服务器硬件，即可拥有应对爆发性流量的能力 20。



### 4. 微软云计算 Microsoft Azure

1. Windows Azure 的计算服务和存储服务

**(1) 计算服务 (Compute Services)**

Windows Azure 的计算服务主要基于“角色（Role）”的概念，将应用程序部署在虚拟机实例中：

- **Web Role（Web 角色）**：
  - **特点**：配置了 **IIS** (Internet Information Services)。
  - **用途**：主要用于运行面向用户的 Web 应用程序，处理 HTTP/HTTPS 请求。
- **Worker Role（辅助角色）**：
  - **特点**：通常**不配置 IIS**。
  - **用途**：用于处理后台任务、密集型计算或长时间运行的工作负载，通过队列与 Web Role 通信。
- **VM Role（虚拟机角色）**：
  - **特点**：允许用户通过 VHD 文件上传和运行自定义的 Windows Server 操作系统镜像，提供更高的控制权。

**(2) 存储服务 (Storage Services)**

Azure 存储服务提供高可用、可扩展的非关系型存储，包含三个核心层次：**前端层、分区层和文件流层**。主要存储类型包括：

- **BLOB (二进制大对象)**：用于存储大型非结构化数据，如视频、图片和备份文件。
- **Table (表)**：提供 NoSQL 结构化存储，适用于存储非关系型的大量数据集，支持快速缩放。
- **Queue (队列)**：提供可靠的消息传递机制，用于计算角色（如 Web Role 和 Worker Role）之间的解耦和异步通信。
- **Drive (驱动器)**：允许应用程序将 BLOB 挂载为 NTFS 卷，方便传统应用的迁移。

------

2. Azure Service Fabric 的体系结构

Azure Service Fabric 是一个分布式系统平台，旨在简化微服务和容器的打包、部署及管理。它采用**分层子系统架构**：

- **传输子系统 (Transport Subsystem)**：
  - 负责实现点对点的数据报信道。
  - 支持集群内部节点之间、以及集群与客户端之间的安全通信。
- **联合子系统 (Federation Subsystem)**：
  - 将物理/虚拟节点逻辑上拼结为一个统一的群集。
  - 通过分布式哈希表（DHT）等机制实现节点发现、故障检测和领导者选举。
- **可靠性子系统 (Reliability Subsystem)**：
  - 负责管理服务的复制、故障转移和负载均衡。
  - 包含“复制器（Replicator）”和“故障转移管理器（Failover Manager）”，确保有状态服务的强一致性和高可用。
- **管理子系统 (Management Subsystem)**：
  - 负责应用程序的完整生命周期管理。
  - 处理部署、升级、健康监测以及服务包的管理。
- **测试能力子系统 (Testability Subsystem)**：
  - 允许开发者模拟节点故障或服务故障，通过“故障注入”测试系统的可靠性。

7. 利用 Visual Studio 开发并部署应用到 Azure 的步骤

根据讲义中关于 Windows Azure 计算服务和开发环境的描述，开发与部署的典型流程如下：

1. **环境准备**：在本地安装 **Visual Studio**，并集成 **Azure SDK**（提供了项目模板、模拟器和库）。
2. **创建项目**：打开 Visual Studio，选择“新建项目”，在云（Cloud）分类下选择 **Azure Cloud Service** 模板。
3. **定义角色 (Roles)**：根据需求添加 **Web Role**（处理 HTTP 请求，配置 IIS）或 **Worker Role**（后台处理任务，不配置 IIS）。
4. **本地调试**：使用 **Windows Azure 模拟器**在本地环境运行和测试应用程序，模拟云端的计算和存储行为。
5. **打包项目**：调试完成后，通过 Visual Studio 生成服务包（.cspkg）和服务配置文件（.cscfg）。
6. **部署上线**：
   - **方式 A**：登录 Azure 管理控制台，上传打包好的文件。
   - **方式 B**：直接在 Visual Studio 中点击“发布（Publish）”，通过授权证书将应用直接推送到 Azure 数据中心。

8. Microsoft Azure 是如何支持大数据处理的？

讲义中提到 Azure 通过多种服务协作来构建大数据处理生态：

- **计算与分析能力**：
  - **HDInsight**：这是 Azure 核心的大数据服务，基于 Apache Hadoop 架构，支持 MapReduce、Hive、Spark 等工具，能够处理极大规模的数据集。
  - **云服务/虚拟机**：提供高性能的虚拟机（PaaS 或 IaaS 模式），支持多层方案和自动化部署，可根据计算需求进行“无限”缩放。
- **存储支持**：
  - **Azure Storage**：通过 **BLOB** 存储海量非结构化数据，通过 **Table** 存储结构化但不复杂的数据。
  - **SQL Azure**：作为云端关系数据库，通过“数据同步”技术支持分布式数据访问和高可用。
- **生命周期与编排**：
  - **Azure Service Fabric**：支持打包、部署和管理可缩放的微服务与容器，能够几秒内高密度部署应用。
  - **Marketplace**：提供第三方数据和应用程序，帮助开发者获取处理大数据所需的特定算法或数据集。
- **辅助工具**：利用 **SQL Azure Reporting** 进行数据报表展现，以及 **Windows Azure CDN** 实现全球范围内的高速数据分发。

### 5. Hadoop3.0：主流开源云架构

1. Hadoop 3.0 的组成

Hadoop 3.0 依然延续了其经典的核心架构，主要由以下四个模块组成：

- **Hadoop Common**：为其他 Hadoop 模块提供基础设施支持的 Java 库和实用工具。
- **HDFS (Hadoop Distributed File System)**：分布式文件系统，负责提供高吞吐量的数据存储，能够运行在廉价的硬件集群上。
- **Hadoop YARN (Yet Another Resource Negotiator)**：资源管理调度框架，负责集群节点的资源管理和作业任务的调度。
- **Hadoop MapReduce**：基于 YARN 的分布式并行计算框架，用于对大规模数据集进行并行处理。

2. 单独部署 ZooKeeper 时，部署 HBase 和 Kafka 的区别

在分布式系统中，HBase 和 Kafka 都高度依赖 ZooKeeper，但它们在**依赖程度**、**存储内容**以及**配置方式**上存在显著区别：

**(1) 对 ZooKeeper 的依赖深度与作用**

- **HBase**：
  - **深度依赖**：ZooKeeper 是 HBase 的核心协调者。HBase 无法在没有 ZooKeeper 的情况下运行。
  - **作用**：用于选举 Master、监控 RegionServer 状态、存储 Meta 表的位置信息以及维护集群的元数据状态。
- **Kafka**：
  - **控制面依赖**：ZooKeeper 主要负责 Kafka 集群的元数据管理。
  - **作用**：用于选举 Controller、管理 Broker 注册信息、存储 Topic 的配置和分区副本（Replica）的状态信息。
  - *注：随着 Kafka 版本的演进（KRaft 模式），其对 ZooKeeper 的依赖正在逐步被移除。*

**(2) 部署时的配置区别**

- **HBase 部署**：
  - 需要在 `hbase-site.xml` 中明确指定 `hbase.zookeeper.quorum`（ZooKeeper 服务器列表）。
  - 需要配置 `hbase.cluster.distributed` 为 `true`。
  - 部署时，HBase 会在 ZooKeeper 中创建一个根目录（默认是 `/hbase`），用于存放其所有的协调数据。
- **Kafka 部署**：
  - 需要在 `server.properties` 中配置 `zookeeper.connect` 属性。
  - 为了安全和隔离，通常建议为 Kafka 在 ZooKeeper 中指定一个特定的路径（如 `localhost:2181/kafka`）。
  - Kafka 部署更强调 Broker ID 与 ZooKeeper 中记录的一致性。

**(3) 数据持久性与交互**

- **HBase** 将关键的系统状态（如集群 Master 到底是谁）实时同步到 ZooKeeper，如果 ZooKeeper 宕机，HBase 集群将立即陷入不可用状态。
- **Kafka** 客户端（旧版本）会直接连接 ZooKeeper 来获取元数据，而新版本客户端更多地通过 Broker 获取信息，ZooKeeper 更多地作为 Broker 之间的内部管理工具。



### 6. 虚拟化技术

1. 虚拟化的目的、与云计算的区别及其关键作用

- **虚拟化的目的**：
  - **资源整合**：打破硬件物理边界，提高硬件利用率（消除“服务器孤岛”）。
  - **隔离与安全**：确保同一物理机上的不同虚拟机（VM）互不干扰。
  - **灵活性**：实现业务的快速部署、动态迁移和容灾备份。
- **虚拟化与云计算的区别**：
  - **虚拟化是技术（工具）**：它是一种资源管理手段，侧重于“把一变多”或“把多变一”。
  - **云计算是模式（服务）**：它是一种商业交付模式，侧重于“按需自助服务”、“弹性伸缩”和“计量计费”。
  - *比喻：虚拟化是高效的发动机，云计算是租车服务。有发动机不代表就是租车公司，但租车公司需要发动机。*
- **关键作用**：
  - **资源池化**：通过虚拟化将底层硬件抽象成计算、存储、网络资源池。
  - **多租户隔离**：保证云端不同用户的数据和运行环境安全。
  - **动态调度**：云计算的弹性（Scale-out/in）依赖于虚拟机的快速创建和热迁移。

2. 比较 VMware、Xen 等产品及对云的支持

   | **产品**          | **核心技术特点**                                             | **对云计算的支持**                                           |
   | ----------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
   | **VMware (ESXi)** | **全虚拟化**的领头羊。硬件兼容性极佳，管理工具（vCenter）极其成熟。 | 提供高度自动化的私有云方案，支持完善的热迁移（vMotion）和高可用（HA）。 |
   | **Xen**           | **半虚拟化 (Para-virtualization)** 的先驱。性能损耗极低，曾是公有云（如早期的AWS）的首选。 | 架构开源，安全性高（特权域隔离），适合大规模多租户的公有云环境。 |
   | **KVM**           | **内核级虚拟化**。直接集成在 Linux 内核中，简单高效，正逐渐取代 Xen。 | 是 **OpenStack** 默认且最完美支持的 Hypervisor，是开源云的主流。 |

3. 计算、存储和网络虚拟化的主要实现方式

- **计算虚拟化**：
  - **CPU虚拟化**：通过指令截获和模拟（如 Intel VT-x 技术）。
  - **内存虚拟化**：引入“客户机物理地址”到“宿主机物理地址”的映射（EPT/NPT技术）。
- **存储虚拟化**：
  - **基于主机**：通过逻辑卷管理（LVM）。
  - **基于存储设备**：通过阵列控制器的冗余技术。
  - **基于网络（分布式）**：通过软件定义存储（如 Ceph），将多台服务器硬盘聚合成存储池。
- **网络虚拟化**：
  - **软件交换机 (vSwitch)**：在服务器内部模拟交换机。
  - **隧道技术 (Overlay)**：如 **VXLAN**，在现有的三层网络上构建二层逻辑网络，解决虚拟机跨三层迁移问题。

4. 讨论桌面虚拟化的实现和作用

- **实现方式 (VDI)**：
  - **VDI (虚拟桌面架构)**：所有运算在数据中心完成，终端只负责接收显示画面和传输输入指令（类似于“看视频”）。
  - **核心协议**：如 PCoIP、RDP 或 ICA，决定了桌面的流畅度。
- **作用**：
  - **数据安全**：数据不落地，所有文件存在服务器，不怕终端丢失。
  - **移动办公**：用户在任何地点、任何设备登录都能接续之前的桌面环境。
  - **统一维护**：管理员一键升级千台电脑的镜像，无需逐台操作。

5. OpenStack 是什么？

OpenStack 是一个**开源的云计算管理平台（Cloud OS）**。

- **它的本质**：它本身**不生产**虚拟化技术（它不包含 Hypervisor），而是通过一系列 API 来**管理**各种虚拟化资源（KVM, VMware, Ceph, Neutron 等）。
- **核心比喻**：它是数据中心的“操作系统”。它向上为用户提供操作界面和 API，向下调度和指挥各种硬件资源。
- **关键架构思想**：高度模块化、松耦合。每个组件（如 Nova, Glance, Keystone）都可以独立运行或组合使用。

6. 总结 OpenStack 的主要组件及其功能

OpenStack 采用微服务架构，每个组件负责一种资源管理。你可以将其核心组件总结为“**五个核心+一个界面**”：

- **Keystone (身份管理)**：**“掌门人”**。负责所有组件的身份验证、令牌管理和服务目录维护。
- **Nova (计算管理)**：**“大脑”**。负责虚拟机的生命周期管理（调度、创建、启停）。它不直接操作虚拟化，而是调用 KVM 等 Hypervisor。
- **Neutron (网络管理)**：**“联络员”**。为虚拟机提供 IP、子网、路由、安全组等网络连接功能。
- **Glance (镜像管理)**：**“仓库管理员”**。提供虚拟机磁盘镜像的存储与查询，创建 VM 时从这里取模板。
- **Cinder (块存储)**：**“外挂硬盘”**。为 VM 提供持久化的块存储空间，VM 删掉后数据仍可保留。
- **Horizon (仪表盘)**：**“窗口”**。即图形化 Web 界面，用户通过点鼠标就能操作上述所有复杂组件。

------

7. OpenStack 各服务模块之间如何协同工作

理解协同工作的关键在于 **API 调用** 和 **消息队列 (RabbitMQ)**。以“**创建一个虚拟机**”的过程为例：

1. **用户**通过 **Horizon** 发出请求。
2. **Keystone** 验证用户权限，发放 Token。
3. **Nova** 接到请求，先去 **Glance** 找镜像，再去 **Neutron** 申请网络地址。
4. **Nova** 的调度器（Scheduler）根据各物理节点资源情况，选出一台合适的物理机。
5. **Nova** 指令该机器上的 Hypervisor（如 KVM）启动 VM。
6. 如果需要持久化存储，**Cinder** 挂载云硬盘到该 VM。
7. 所有操作状态通过 **消息队列** 实时同步。

这是一道发散思维题，可以从以下三个维度概括：

| **维度**     | **OpenStack**                                                | **AWS (Amazon Web Services)**                            |
| ------------ | ------------------------------------------------------------ | -------------------------------------------------------- |
| **部署模式** | **私有云**为主。用户在自己的机房搭建，拥有完全控制权。       | **公有云**。资源属于亚马逊，用户按需租用。               |
| **性质**     | **开源软件**。免费使用源代码，但需要强大的技术团队进行运维。 | **专有服务**。闭源，用户只需使用 API，无需操心底层维护。 |

9. 对比超融合架构 (HCI) 和传统架构的优缺点

- **传统架构 (3-Tier)**：服务器 + 交换机 + 独立的存储阵列 (SAN/NAS)。
  - **优点**：**高性能、高可靠**。存储和计算解耦，升级其中一方不影响另一方。
  - **缺点**：**复杂、昂贵**。需要专业的存储工程师维护，且无法轻松实现横向扩展（Scale-out）。
- **超融合架构 (HCI)**：计算、存储集成在同一台服务器内，通过软件定义存储。
  - **优点**：
    1. **极简性**：部署非常快，开箱即用。
    2. **线性扩展**：加一台服务器就等于增加了计算+存储能力。
    3. **成本低**：不需要昂贵的专用存储光纤交换机和阵列。
  - **缺点**：
    1. **资源绑定**：如果只想扩容存储，必须连带计算资源一起买，造成资源浪费。
    2. **性能瓶颈**：由于存储程序占用了服务器本来的 CPU/内存，重负载下性能可能略逊于顶级独立存储。

### 7. 容器技术

1. 容器的概念及与虚拟机的区别

**容器概念**：容器是一种轻量级操作系统层面的虚拟机，它为应用软件及其依赖组件提供了一个资源独立的运行环境。它通过容器引擎将应用及其依赖包打包到可移植的镜像中，实现应用与操作系统的解耦 。

**区别对比** ：

- **架构**：虚拟机拥有独立的Guest OS，隔离性强；容器共享宿主机内核和OS，隔离性相对较弱。
- **性能**：虚拟化有性能损耗（>15%）；容器计算/存储几乎无损耗。
- **启动速度**：虚拟机为分钟级；容器为秒级。
- **资源占用**：虚拟机镜像大（GB级），单机运行数量少（10~100个）；容器镜像小（MB级），单机密度高（1000+个）。



2. 容器化的关键技术及其优势是什么?

- **关键技术** ：
  - **Namespace（名称空间）**：实现进程、网络、挂载点、UTS、IPC及用户的隔离。
  - **Cgroups（控制组）**：实现资源（CPU、内存、I/O）的限制、记录和隔离。
  - **Chroot**：改变程序运行参考的根目录位置，实现文件系统隔离。
  - **联合文件系统（UnionFS）**：实现镜像的分层存放和写时复制机制。
- **优势** ：
  - **敏捷开发与部署**：一次构建、到处运行，秒级创建启动。
  - **环境一致性**：封装了应用依赖及系统信息，计算环境可移植。
  - **弹性伸缩**：由编排工具管理，可短时间内自动完成扩容。
  - **高效率**：资源利用率高，成本低。

3. 简述 Docker 的系统架构及基本使用流程

- **系统架构**：Docker主要由以下五个部分组成 ：
  - **Docker Client（客户端）**：用户操作接口。
  - **Docker Daemon（守护进程）**：运行在宿主机上，管理容器、镜像。
  - **Docker Image（镜像）**：容器的只读模板。
  - **Docker Container（容器）**：镜像运行后的实例。
  - **Docker Registry（仓库）**：存储和分发镜像的服务器（如Docker Hub）。
- **基本使用流程** ：
  1. 应用程序开发。
  2. 编写 **Dockerfile**。
  3. **Build**：构建Docker镜像。
  4. **Push**：发布镜像到仓库。
  5. **Pull & Run**：下载并运行镜像。

4. Rkt 容器与 Docker 有哪些差别?

- **安全性**：Rkt 改进了 Docker 在安全方面的缺陷，支持镜像签名验证 。
- **架构设计**：Rkt 遵循 **AppC** (Application Container Specification) 规范，强调组件式工具、去中心化和开放性标准，而 Docker 当时更多是单体式设计 。
- **权限管理**：Docker 默认在宿主机 root 权限下运行（存在风险），Rkt 更强调运行时的隔离与安全性 。

5. Garden 容器是如何实现资源控制的?

- Garden 容器主要通过 **Cgroups（控制组）** 来实现资源控制 。
- 具体方式包括：
  - **资源限制**：设定 CPU 使用配额和内存使用上限 。
  - **优先级控制**：为不同的控制族群分配不同的资源使用优先级 。
  - **资源统计**：监测和测量系统资源的使用量 。
  - **进程控制**：对容器内的任务进行冻结或启动等操作 。

6. 简述 Kubernetes 的系统架构组成

​	Kubernetes（K8s）是一个用于自动部署、扩展和管理容器化应用程序的开源系统 11。其架构通常由 **Master 节点** 和 **Worker 节点**（也称为 Node）组成：

- **Master 节点（控制平面）**：负责集群的管理。
  - **API Server**：集群的控制入口，提供 RESTful 接口。
  - **Scheduler**：资源调度器，决定将 Pod 放在哪个 Node 上。
  - **Controller Manager**：管理控制器，确保集群处于期望状态。
  - **etcd**：存储集群所有状态的分布式键值数据库。
- **Worker 节点**：运行实际的应用容器。
  - **Kubelet**：负责维护容器的生命周期，与 Master 节点通信。
  - **Kube-proxy**：负责集群内部的网络代理和负载均衡。
  - **Runtime**：容器运行时环境（如 Docker、containerd）。

7. 主流容器编排系统有哪些？它们的特点是什么？

| **编排系统**         | **开发/主导者** | **特点**                                                     |
| -------------------- | --------------- | ------------------------------------------------------------ |
| **Kubernetes (K8s)** | Google          | 事实上的行业标准。具备强大的自动化调度、服务发现、负载均衡和弹性伸缩能力，生态极其丰富。 |
| **Docker Swarm**     | Docker 公司     | 原生集成在 Docker 中，上手简单，轻量级，但功能广度（如自我修复、复杂拓扑支持）弱于 K8s。 |
| **Apache Mesos**     | Apache 基金会   | 侧重于两层调度和大规模扩展（支持 10000+ 节点），常用于混合运行大数据（Hadoop）与容器化负载 3。 |
| **CoreOS Fleet**     | CoreOS          | 对 CoreOS 集群进行控制的工具，具备高可靠的管理能力，但目前已逐渐边缘化 4。 |

8. Kubernetes 采用了哪些理念来满足大规模、大数据计算场景？

​	K8s 通过以下核心理念应对大规模和高复杂度的计算场景：

- **资源调度（Resource Scheduling）**：集中化管理和调度资源，按需为容器分配 CPU、内存，确保大规模任务的最优布局。
- **服务发现与负载均衡**：通过注册中心自动感知服务变化，当压力增加时自动扩展并均衡流量，避免单点过载。
- **弹性伸缩（Elastic Scaling）**：根据监控指标自动增减 Pod 实例或 Node 主机资源。
- **声明式 API（Declarative API）**：通过 YAML 描述“最终状态”，由系统自动收敛，降低了运维大规模集群的复杂度。
- **Pod 概念**：将多个紧耦合的容器打包在 Pod 中，共享网络和存储，适配了大数据处理中常见的边车（Sidecar）模式。

9. 如何基于 Docker 和 Kubernetes 来构建一个 Hadoop 大数据系统？

1. **镜像化 (Dockerization)**：将 Hadoop 的各个组件（NameNode, DataNode, ResourceManager, NodeManager）分别编写 **Dockerfile**。
   - 在镜像中预装 JDK、Hadoop 包，并配置好环境变量和启动脚本。
2. **网络与存储配置**：
   - 使用 K8s 的 **Service** 解决组件间的固定访问地址问题（服务发现）。
   - 利用 **PersistentVolume (PV)** 和 **PVC** 挂载持久化数据卷，确保 DataNode 数据在容器重启后不丢失。
3. **编排部署 (Kubernetes Orchestration)**：
   - 编写 **YAML 编排文件**。对于有状态的大数据组件，通常使用 **StatefulSet** 部署（保证 Pod 有序启动和持久化标识）。
   - 通过 K8s 的调度策略（如亲和性 Affinity）将 DataNode 分散在不同的物理节点上。
4. **自动化运维**：利用 K8s 的监控能力和热迁移技术，调节大数据处理的峰谷压力，降低应用延迟。

### 8. 云原生技术

1. 云原生的概念及核心内容是什么？

- **云原生的概念**：云原生（Cloud Native）由 Pivotal 公司的 Matt Stine 在 2013 年提出，其应用具有**容器化包装**、**动态管理**和**微服务化**三大特征
- **核心内容**：包含微服务（MicroServices）、敏捷基础设施（Agile Infrastructure）、DevOps 和持续交付（Continuous Delivery）四大部分 。

2. 什么是微服务？微服务架构作用及其分类

- **概念**：微服务是将大型复杂软件拆分成多个简单应用，每个应用描述一个小业务且可独立部署。它遵循 UNIX 哲学——“每种服务只做一件事”。
- **作用**：
  - **降低复杂度**：业务清晰，代码量少，易于维护
  - **独立性**：支持独立部署和扩展，局部修改不影响整体。
  - **灵活性**：技术栈不受限，启动速度快，可按需细粒度伸缩 
- **分类（架构实践形式）**：
  - **侵入式架构**：服务框架嵌入代码，实现类继承，如 **Spring Cloud** 
  - **非侵入式架构**：以代理形式接管网络，对应用透明，如 **Service Mesh（服务网格）** 

3. 对比分析 Spring Cloud 和 Dubbo 架构特点

根据讲义内容，两者的主要对比点如下：

| **特点**       | **Spring Cloud**                                             | **Dubbo**                                             |
| -------------- | ------------------------------------------------------------ | ----------------------------------------------------- |
| **定义与定位** | 基于 Spring Boot 的一整套微服务实现框架。                    | 阿里开源的高性能 RPC 服务框架。                       |
| **核心能力**   | 包含服务发现 (Eureka)、配置中心 (Config)、总线 (Bus)、链路追踪 (Sleuth) 等。 | 侧重面向接口代理的 RPC 调用、负载均衡、服务自动发现。 |

4. 如何基于容器技术构建容器云平台？

1. **资源分割与调度**：以容器作为资源的基本单位，实现资源共享与隔离。

2. **核心平台层**：

   - **容器引擎**：如 Docker，负责应用打包与运行。
   - **编排调度系统**：使用 **Kubernetes (K8s)** 或 Swarm 实现对 Pod（容器组）的自动化管理。

3. **服务支持层**：

   - **基础服务**：集成网关、API 服务、控制服务及构建服务。
   - **中间件集成**：接入分布式消息队列（Kafka/RabbitMQ）和缓存服务（Redis） 

4. **运维监控体系**：

   - **日志与报警**：集成 Prometheus、Zipkin 等进行链路跟踪和实时监控 
   - **存储与网络**：通过 Cinder (块存储) 和 Neutron (网络) 等底层技术支持。

5. **交付流水线**：结合 **DevOps** 工具（如 Jenkins, Ansible）实现从代码到容器镜像的持续集成与自动部署 。\

   

5. 消息中间件传输模型有哪两种？

消息中间件（Message Middleware）主要有两种消息传递模型：

- **点对点模型（Point-to-Point, P2P）**：基于**队列（Queue）**。消息生产者将消息发送到队列，消息消费者从队列中提取。每条消息只有一个消费者，一旦被消费，消息即从队列中移除。
- **发布/订阅模型（Publish/Subscribe, Pub/Sub）**：基于**主题（Topic）**。消息发布者将消息发布到某个主题，多个订阅了该主题的消费者都可以接收到该消息。这种模型支持“一对多”的通信方式。

7. 什么是 DevOps？常用工具有哪些？

- **概念**：DevOps 是 **Development（开发）** 和 **Operations（运维）** 的组合词。它是一种文化、过程和工具的结合，旨在缩短软件开发生命周期，通过**自动化**手段提高软件交付的质量和频率。
- **核心逻辑**：打破开发人员和运维人员之间的“墙”，实现从代码编写到上线发布的无缝衔接。
- **常用工具**：
  - **代码管理**：Git, GitLab, GitHub。
  - **构建/集成**：**Jenkins**, Travis CI。
  - **容器化**：Docker。
  - **配置管理/自动化运维**：**Ansible**, Puppet, Chef。
  - **监控与日志**：Prometheus, ELK (Elasticsearch, Logstash, Kibana)。

8. 持续交付与持续集成、持续部署的关系

这三者是云原生和 DevOps 中“流水线”的不同阶段，具有递进关系：

1. **持续集成 (Continuous Integration, CI)**：
   - **核心**：开发人员频繁地将代码集成到主干。
   - **流程**：每当有代码提交，系统自动进行**编译、运行单元测试**，确保新代码没有破坏原有功能。
2. **持续交付 (Continuous Delivery, CD)**：
   - **核心**：在持续集成的基础上，将代码部署到**类生产环境**（如测试环境、预发布环境）。
   - **状态**：确保代码随时处于**可发布状态**，但最终发布到生产环境通常需要人工审批或手动触发。
3. **持续部署 (Continuous Deployment, CD)**：
   - **核心**：这是最高级别的自动化。
   - **流程**：通过所有自动化测试的代码，**自动**部署到生产环境，无需人工干预。

**总结关系**：

- **CI** 是基础，解决代码冲突和基础质量问题。
- **持续交付** 扩展了集成，确保代码在技术上是可靠且可发布的。
- **持续部署** 是最终目标，实现业务价值的最快交付。

### 9. 云计算数据中心

1. 集装箱数据中心有哪些优点？常见的节能措施有哪些？

- **优点**：
  - **高密度**：在有限的空间内集成大量的服务器、存储和网络设备。
  - **低成本**：采用标准化组件，工厂预制，减少了现场建设的时间和人力成本。
  - **可移动/易部署**：像集装箱一样易于运输，可以快速部署到全球任何地方，实现“即插即用”。
  - **按需扩展**：支持模块化扩展，企业可根据业务需求逐个增加集装箱。
- **常见节能措施**：
  - **精确制冷**：采用冷热通道隔离技术，减少冷气浪费。
  - **自然冷却**：利用室外自然空气（风冷）或水资源进行散热，减少空调能耗。
  - **高效供电**：使用高效率的UPS和配电系统，减少电能传输过程中的损耗。

2. 云计算数据中心配电系统节能的原理是什么？

配电系统节能的核心原理在于**减少转换层次**和**提高转换效率**：

- **简化拓扑结构**：通过减少变压和整流的中间步骤，降低电能在传输和转换过程中的热损耗。
- **高压直流供电 (HVDC)**：传统的配电使用交流电（AC），需要多次“交流-直流”转换。采用直流供电可以直接为服务器供电，减少逆变环节，效率可提升至 90% 以上。
- **谐波抑制**：通过有源滤波等技术减少电力系统中的谐波损耗。
- **智能化管理**：根据IT设备的实际负载动态调整电源模块的运行状态，使其始终工作在高效区间。

3. 能源利用效率 (PUE) 的计算方式是什么？

PUE（Power Usage Effectiveness）是衡量数据中心能效的最关键指标。

- 计算公式：
  $$
  PUE = \frac{数据中心总设备能耗 (Total Facility Power)}{IT 设备能耗 (IT Equipment Power)}
  $$

- **指标含义**：

  - PUE 越接近 **1**，表示非 IT 耗能（如制冷、照明、配电损耗）越少，效率越高。
  - 传统数据中心 PUE 常在 2.0 以上，先进的云数据中心可达到 1.1-1.2 左右。

4. 海底数据中心与传统数据中心对比

| **特性**     | **传统数据中心 (Traditional)**      | **海底数据中心 (Underwater)**                              |
| ------------ | ----------------------------------- | ---------------------------------------------------------- |
| **散热成本** | 需消耗大量电力运行空调及冷却塔。    | **极低**。直接利用海水进行自然循环散热，能效极高。         |
| **部署速度** | 需土地审批、基建，通常耗时 1-2 年。 | **极快**。工厂预制后直接下水，数月内可上线。               |
| **资源消耗** | 占用土地资源，且消耗大量淡水。      | **零土地、零淡水占用**。适合沿海人口密集区。               |
| **维护性**   | 方便人员进入进行硬件维修。          | **基本不维护**。设计为 5-10 年免维护，坏了直接回收。       |
| **可靠性**   | 受环境温度、人为操作影响。          | **更高**。海底温度恒定且充入氮气环境，硬件故障率低于陆地。 |

### 10. 云际计算技术

1. 传统云计算有哪些缺陷和挑战？

传统单云服务主要面临以下四大挑战：

- **平台锁定问题**：消费者过分依赖单一云服务商，由于接口异构，无法在不产生大量切换成本的情况下进行灵活迁移。
- **数据安全性与隐私**：数据全部交给单一供应商，存在隐私泄露风险；一旦云服务商发生故障或倒闭，用户数据将面临丢失风险。
- **服务质量瓶颈**：单云在应对爆发性需求、全域覆盖和资源多样性方面存在局限。
- **资源孤岛**：各云服务商之间缺乏统一的协作标准，导致资源无法跨云流动和共享。

2. 云际计算的概念是什么？有哪些代表性工作？

- **概念**：云际计算（Inter-Cloud Computing）是以自主参与、对等协作、透明服务、价值交换为特征，通过多云协作和资源共担，满足复杂应用对资源爆发性、全域性和多样性需求的新型计算范式。
- **代表性工作**：
  - **IEEE P2302标准**：定义了云际互联的原则、操作及管理方式。
  - **超级云（Supercloud）**：一种跨多个公有云的云架构。
  - **云总线（CloudBus）**：提供资源调度和云际协调功能。
  - **中国科学家的贡献**：梅宏院士等提出云际计算协作模式。

3. 云际计算的优势有哪些？

- **消除厂商锁定**：通过统一标准实现跨云迁移和资源调度。
- **提高可靠性**：利用多云冗余，即使某个云平台失效，业务仍可运行。
- **优化性能与延迟**：根据用户地理位置选择最近的云节点（全域性）。
- **弹性最大化**：能够调用多个云平台的空闲资源应对爆发式流量需求。
- **成本优化**：根据各云服务商的实时报价进行动态切换，降低支出。

4. 云际计算的架构有哪些？

- **多云架构（Multi-cloud）**：
  - **机制**：用户同时使用多个独立的云。
  - **优缺点**：降低风险，但管理复杂度高，数据同步困难。
- **混合云架构（Hybrid Cloud）**：
  - **机制**：结合私有云和公有云。
  - **优缺点**：兼顾安全与弹性，但由于私有云与公有云环境差异大，集成难度高。
- **联邦云架构（Federated Cloud）**：
  - **机制**：各云商通过对等协议自愿组合，资源共享。
  - **优缺点**：资源利用率极高，但需要复杂的信任和结算机制。

5. 云际协作环境的三类核心服务是什么？分别阐述其作用。

- **实体服务（Entity Services）**：
  - **作用**：主要负责云际实体的身份管理、注册和认证，确保参与协作的云服务商或用户的身份真实可信。
- **交互服务（Interaction Services）**：
  - **作用**：负责跨云的通信、协议转换和消息传递，解决不同云平台之间接口不统一（异构性）的障碍。
- **交易服务（Transaction Services）**：
  - **作用**：负责资源计费、价值交换和智能合约管理，确保跨云资源共享时的经济公平性和合规审计。

2. OceanBase、DataCube 与传统关系数据库有何异同？

- **OceanBase (分布式数据库)**：
  - **异同**：不同于传统数据库的单机架构，OceanBase 采用**分布式架构**。它支持跨机房多副本同步（三地五中心），在保持 SQL 兼容和 ACID 特性的同时，解决了传统数据库在大规模并发下的扩展性瓶颈。
- **数据立方 DataCube (分布式计算/存储)**：
  - **异同**：基于分布式列存储技术，主要针对**海量数据秒级查询**优化。传统数据库在处理百亿级数据关联查询时速度慢，DataCube 利用索引和分布式并行计算实现极速响应。
- **总结**：传统数据库胜在**事务强一致性**和**通用性**，但在数据规模和高可用性上不如国产分布式数据库。

3. 比较 MapReduce 与 DPCA 的优缺点

- **MapReduce (经典大数据框架)**：
  - **优点**：模型简单（Map+Reduce），适合处理静态、超大规模数据集，容错性强。
  - **缺点**：实时性差，中间结果频繁读写磁盘，不适合流式计算或迭代计算。
- **DPCA (数据并行计算架构/类似 Spark 思想)**：
  - **优点**：通常指更先进的并行处理模式，如内存计算。速度比 MapReduce 快（可达10-100倍），支持更复杂的计算拓扑。
  - **缺点**：对内存资源消耗较大。

4. 阿里云 ECS 使用场景及与传统服务器相比的优势

- **使用场景**：**电商促销（如双11）**。平时业务量平稳，只需少量服务器；促销开始前，几分钟内扩容出上万台 ECS 应对高并发流量。
- **对比优势**：
  - **弹性伸缩**：云服务器秒级升降配，传统服务器需采购、上架，周期长达数周。
  - **成本**：按需付费，无需为峰值流量预留闲置硬件。
  - **免运维**：云厂商负责底层硬件维护、电力和散热，用户只管业务。
  - **高可用**：云平台提供镜像备份和自动迁移，硬件损坏不影响业务运行。

6. 简述云计算技术发展概况

- **第一阶段（起步）**：以“BAT”为代表的企业开始自研底层的分布式文件系统和虚拟化技术。
- **第二阶段（成熟）**：云计算从互联网行业向金融、政务、制造等传统行业渗透。容器技术、微服务和云原生架构成为主流。
- **第三阶段（智能化/云际）**：云与 AI、边缘计算深度融合。发展趋势包括：
  1. **标准化**：如 NIST 参考模型和 SNIA 云存储标准。
  2. **混合云与多云**：企业倾向于不绑定单一厂商，追求跨云协作。
  3. **云际计算**：解决“云孤岛”问题，实现跨服务商的资源流动与对等协作。
